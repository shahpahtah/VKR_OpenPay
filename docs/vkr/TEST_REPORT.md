# Отчет тестирования OpenPay

Дата прогона: 28.04.2026.

## Итог

Проект собирается без ошибок и предупреждений, автотесты проходят, ключевые web-страницы открываются по ролям, основной платежный сценарий проходит до отправки в банк-адаптер. Интеграция с T-Банк Sandbox дополнительно проверена из пользовательской PowerShell-сессии и через OpenPay.

## Исправлено во время тестирования

| Область | Проблема | Исправление |
|---|---|---|
| `/Admin/ApprovalRoutes` | Страница падала с `FormatException`: SQLite/EF пытался сортировать `decimal` и парсить `100000.01` при русской культуре | Сортировка маршрутов перенесена в память после загрузки DTO |
| Запуск web-приложения в текущей среде | Kestrel пытался писать в Windows EventLog без прав, запросы падали | В `Program.cs` явно оставлены `Console` и `Debug` logging providers |
| ASP.NET Data Protection | Ключи пытались читаться из `AppData`, доступ был запрещен | Ключи Data Protection перенесены в `OpenPay.Web/DataProtection-Keys` |
| UI-полировка | В CSS оставались крупные радиусы карточек и элементов управления | Радиусы карточек, форм и кнопок приведены к более строгому рабочему стилю |
| Предупреждения NuGet | `OpenPay.Infrastructure` содержал лишние зависимости `Microsoft.Extensions.Hosting` и `Microsoft.AspNetCore.Http.Abstractions` | Зависимости удалены, сборка проходит без предупреждений |

## E2E платеж

Фактический сценарий:

1. Бухгалтер создал тестовый платеж.
2. При повторе с тем же контрагентом, датой и суммой система вернула ошибку возможного дубля. Это ожидаемо.
3. Для уникального платежа был выполнен переход на согласование.
4. Руководитель увидел документ в очереди согласований.
5. Руководитель утвердил документ.
6. Платеж получил демо-подпись.
7. Итоговый статус при демо-адаптере: `Отправлен в банк`.
8. При подключении T-Банк Sandbox и проверке статуса итоговый статус: `Исполнен`.

Замечание: HTML-скрипт не всегда надежно распознает промежуточный текст статуса из-за кодировки/разметки, поэтому функциональный факт подтверждался через наличие платежа в очереди руководителя и финальный статус.

## Выписки и сверка

Фактический сценарий:

1. В текущей dev-базе были счета без банковского подключения.
2. При попытке загрузить выписку по такому счету система вернула понятную ошибку: для счета не настроено банковское подключение.
3. Затем было создано демо-подключение `MOCK`.
4. Был создан счет организации с привязкой к этому подключению.
5. Загрузка выписки по счету завершилась успешно.
6. Страница показала результат сверки.

## T-Банк Sandbox

В коде есть адаптер `TBankSandboxAdapter`, регистрация в DI и настройки URL песочницы.

Проверенные endpoint-ы:

- `GET /api/v1/bank-statement` вернул `200 OK` и JSON выписки.
- `POST /secured/api/v1/payment/ruble-transfer/pay` вернул `201 Created`.
- `GET /secured/api/v1/payment/{paymentId}` вернул статус `EXECUTED`.

Через интерфейс OpenPay платеж получил демо-подпись, был отправлен в T-Банк Sandbox, получил `BankReferenceId` и перешел в статус `Исполнен`.

## Остаточные замечания

- Browser-based E2E через in-app browser не выполнен: Node REPL в среде Codex вернул `Access denied`. Вместо этого выполнены HTTP smoke/E2E проверки.
- Для стабильной сборки на текущем preview SDK использовался `/m:1 /p:BuildInParallel=false`. Без этого MSBuild иногда завершался с `exit code 1` без ошибок.
- Локальная dev-база содержит тестовые данные, созданные во время проверки. Для чистой демонстрации лучше удалить `OpenPay.Web/openpay-dev.db*` и дать приложению заново выполнить seed.
